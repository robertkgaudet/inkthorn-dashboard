How a Website Actually Works

Every time you visit a website, a small miracle happens in under a second. Here's exactly what's going on behind the scenes.

🧑‍💻
Step 1
You Type a URL
You type textos.ai in your browser. That's a human-readable address — like a street address for a computer.
→
📖
Step 2
DNS Lookup
DNS translates the name into an IP address (like 104.21.8.43). Think of it as the internet's phone book — names to numbers.
→
🌐
Step 3
Request Travels
Your request hops across the internet in milliseconds, reaching the nearest server that hosts the site — often Cloudflare.
→
🖥️
Step 4
Web Server Responds
A web server — a computer that never turns off — receives your request and sends back files: HTML, CSS, and JavaScript.
→
🎨
Step 5
Browser Renders
Your browser reads those files and paints the page you see. This whole process typically takes under 200 milliseconds.
HTML — Structure

The skeleton of the page. Defines what's on it: headings, paragraphs, buttons, images. Like the walls and rooms of a building.

CSS — Style

The paint, furniture, and lighting. CSS makes things look the way they look — colors, fonts, spacing, layout, animations.

JavaScript — Behavior

The light switches and elevators. JS makes the page interactive — handles clicks, loads data, updates content without reloading.

A real application has six distinct layers. Each does a specific job. When you know them, you can direct AI to build any of them.

🎨
UX / Interface Layer
What users see and interact with — HTML structure, CSS styling, JavaScript behavior. This is the only layer users ever experience directly.
Astro Framework
🗺️

Maps URLs to pages. When a user visits /dashboard, routing says "show them the dashboard." Without routing, every URL would show the same blank page.
Astro + _redirects
⚙️
Backend / API Layer
The brains — processes form submissions, calls the AI, handles payments, enforces business logic. Invisible to users, essential to everything.
Cloudflare Workers
🔐
Authentication Layer
The bouncer — knows who you are and what you're allowed to see. Handles sign-up, login, logout, "forgot password," and keeps each user's data private.
Supabase Auth
🗄️
Database Layer
The permanent memory. Anything that needs to survive a page refresh — user data, posts, settings, transactions — lives here in structured tables.
Supabase / Postgres
🌍
Hosting / CDN Layer
Where your files live and how they reach users globally. CDN = Content Delivery Network. Copies of your app sit on servers near every user on earth.
Cloudflare Pages

Module 01 — Technical Foundation

Cloudflare — The Internet's Front Door

When someone visits your app, the very first computer they connect to is likely a Cloudflare server — not your origin server. That's by design, and it's a superpower.

🌐

Global Edge Network

Cloudflare operates 300+ data centers in 100+ countries. When your user in Tokyo visits your site, they hit a Cloudflare server in Tokyo — not one in Virginia. Result: sub-100ms load times anywhere on earth.

📄

Cloudflare Pages — Static File Hosting

Your HTML, CSS, and JavaScript files are deployed to every edge location automatically. Free SSL certificate included. One deploy command puts your app live globally in under 60 seconds.

⚡

Cloudflare Workers — Serverless Backend

Run backend code (your API, AI calls, payment processing) directly at the edge — no traditional server to manage, no server bills to pay. Code runs in 0ms cold start time.

🛡️

Security Included, Free

Automatic DDoS protection, SSL/TLS certificates, Web Application Firewall, bot filtering. Enterprise-grade security that would cost thousands/month elsewhere — free on Cloudflare's free tier.

  • Cold starts, provisioning time, scaling setup
  • SSL certificates require manual renewal

Module 01 — Technical Foundation

Supabase — Your Database & Login System

Think of Supabase as two things in one: a spreadsheet in the cloud that thousands of users can read and write to simultaneously, plus a complete login system — all managed through a visual dashboard.

🗄️ The Database

PostgreSQL — Industry Standard

Postgres is the same database engine used by Instagram, Reddit, Shopify, and Airbnb. It's been battle-tested for 30+ years and can scale from 10 users to 100 million. Supabase wraps it with a visual table editor that looks like Google Sheets — you can see your data without writing any code.

🔐 Authentication

Complete Identity System, Built In

Sign up, sign in, sign out, "forgot password," magic links, and social login (Google, GitHub, Apple) — all ready to use. Without Supabase, building just the login system would take weeks. With Supabase, it's enabled in an afternoon.

🔒 Row Level Security

Each User Sees Only Their Data

RLS is a database rule that enforces privacy automatically. User A's posts are invisible to User B — not because of code you write, but because the database itself enforces it. This is how multi-user apps work safely at scale.

⚡ Auto-Generated API

Your Database Becomes an API Instantly

The moment you create a table in Supabase, a REST API for that table is automatically generated. Your app can read, write, update, and delete data without you having to write any server code to handle it.

💡

The big picture: Supabase handles everything about "who is this user and what data belongs to them." Cloudflare handles everything about "how does the app reach the user." Together, they replace what used to require a full-time DevOps engineer.

The Two Roles — Architect vs. Implementer

Claude.ai — Your Role
The Architect
  • Write briefs specifying WHAT to build and WHY
  • Define constraints and patterns to follow
  • Reference existing working code as the pattern
  • Make all product decisions
  • Decide when something is good enough to ship
🚫 Never Write specific function bodies, line-by-line fixes, or paste code blocks in your briefs.
Claude Code — AI's Role
The Implementer
  • Figures out HOW to implement your brief
  • Writes and tests the actual code
  • Deploys to the test environment first
  • Reports what was done and what to verify
  • Never makes product decisions on its own
🚫 Never Implement anything without understanding the existing architecture first.

Morning Session — 9:45 AM

Writing Briefs That Actually Work

Good briefs produce good code. Vague briefs produce confused code. Rushed briefs produce wasted hours. This template eliminates all three.

WHAT: One sentence — the outcome, not the implementation "Add a live badge to the nav that pulses when a task is running" FILES: Exact file paths only — limits scope and prevents accidents src/pages/business/live.astro PATTERN: Reference existing working code in the project "Same animation as the pulsing dot in builder.astro around line 847" DO NOT: Explicit exclusions — prevents Claude Code from touching the wrong files "Do not modify builder.astro, manage.astro, or any other page" VERIFY BY: How you will confirm it worked — in your own words "After test deploy, badge appears in nav and pulses when a task is running"
✅

2 sentences — not 30 lines of code. Describe outcomes, not implementations. That is the entire skill.

Module 03 — Architecture

Two Environments — Never Mix Them

Test Environment
your-app-test.pages.dev
  • All deployments go here first — no exceptions
  • No real charges, no real user data
  • Stripe in test mode (fake payment cards)
  • This is where you verify before going live
  • Only your test account can access it
Production Environment
app.yourdomain.com
  • Only after test environment is fully verified
  • Real users, real data, real transactions
  • Requires your explicit approval every time
  • Never deploy here without testing first
  • If it breaks here, real users are affected
🔥

A single typo in a configuration file took down production for 2 hours. The fix was one line. The lesson: test first, every single time.

Afternoon Session — 2:45 PM

The Deploy Ritual — Follow This Order Every Time

1
Pre-deploy authentication check
npx wrangler whoami     node --version
Confirms your API token is loaded. Node must be v22+. Skip this and you get mystery failures.
2
Deploy to the TEST environment
npm run deploy:test
Open the test URL. Sign in as a real user. Click the feature you changed. Check browser console. Test at mobile width (390px).
3
Give explicit approval to go to production
"I verified X, Y, and Z on test. Approving for production." — Write it down. Then and only then proceed.
4
Deploy to PRODUCTION
npm run build    npx wrangler pages deploy dist --project-name your-app
5
Verify on production — same checks as test
Confirm the feature works AND that nothing else broke. Then it's done.
✕
Never: deploy to prod without testing · trust AI's "it works" · bundle multiple changes

Afternoon Session — 1:00 PM

Building Real Features — The 5 Non-Negotiable Rules

🔍

Research First, Always

Search project knowledge and read CLAUDE.md before writing any brief. This single rule would have saved 7 hours on a routing problem that was already documented.

1️⃣

One Change at a Time

Never bundle multiple features in one deploy. One feature, one brief, one deploy. Verify it works. Then move to the next.

✅

Test Before Production — Always

"It worked in test" is the only acceptable green light for production. Never deploy directly to prod to save time.

👁

Verify With Your Own Eyes

Open the browser. Sign in. Click the thing. Read the console. Only what you verify yourself counts as done. Nothing else.

🚫

No Fallbacks, No Placeholder Data

If data is missing, fail loudly. Never substitute "Your Business" for a real business name. Silent failures hide real bugs and mislead real users.

Afternoon Session — 1:45 PM

When Things Break — The Debugging Brief Formula

STEP 1 — DO NOT FIX ANYTHING YET Read the exact error message. Paste it here in full: [error text] Read the file where the error occurs. Show me lines X through Y. STEP 2 — TRACE THE CODE PATH What function called what function that caused this error? Show me the exact call chain, step by step. STEP 3 — SEARCH BEFORE FIXING Search the entire codebase for the same pattern. How many places have the same potential problem? STEP 4 — ONLY THEN APPLY A FIX Fix the root cause — not the symptom. Show me the exact changes (diff) before applying anything. Deploy to test and verify the error is completely resolved.
📖

Real story: A 7-hour debugging session was caused by skipping Step 1. We guessed at solutions and deployed 15+ times. The answer was a single line in a config file — already documented in CLAUDE.md. Research before fix. Every time.
