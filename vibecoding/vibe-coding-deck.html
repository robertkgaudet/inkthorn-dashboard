Morning Session — 9:30 AM

The Two Roles — The Most Important Concept

CLAUDE (claude.ai)
You — The Architect
  • Write briefs specifying WHAT and WHY
  • Define constraints + patterns to follow
  • Reference existing working code as the pattern
  • Decide when something is good enough to ship
🚫 Never Write specific function bodies, line-by-line fixes, or copy-paste code blocks
CLAUDE CODE (CLI)
The Implementer
  • Figures out HOW to implement the brief
  • Writes and tests the actual code
  • Deploys to test environment first
  • Reports what was done + what to verify
🚫 Never Implement without understanding the existing architecture first

Morning Session — 9:45 AM

Writing Briefs That Work

Good briefs produce good code. Vague briefs produce confused code. Rushed briefs produce wasted hours.

WHAT: One sentence — the outcome, not the implementation "Add a live badge to the nav that pulses green when a task is running" FILES: Exact file paths — limits scope src/pages/business/live.astro PATTERN: Reference existing working code "Same animation as the pulsing dot in builder.astro ~line 847" DO NOT: Explicit exclusions prevent accidents "Do not modify builder.astro, manage.astro, or any other page" VERIFY BY: How to confirm it worked "After test deploy, confirm badge appears and pulses when tasks run"
💡

2 sentences, not 30 lines of code. Describe outcomes, not implementations. That's it.

Module 03 — Architecture

Two Environments — Never Mix Them

TEST ENVIRONMENT
your-app-test.pages.dev
  • All deployments go here first
  • No real charges, no real data
  • Stripe in test mode (fake cards)
  • Where you verify before going live
  • Your test account only
PRODUCTION ENVIRONMENT
app.yourdomain.com
  • Only after test is verified
  • Real users, real data, real money
  • Requires your explicit approval
  • NEVER deploy without testing first
  • If it breaks here, users are affected
🔥

A single typo in a .env file took down production for 2 hours. The fix was one line. Test first. Always.

Afternoon Session — 2:45 PM

The Deploy Ritual — Do This In Order Every Time

1
Pre-deploy auth check
npx wrangler whoami    node --version
Confirms token is loaded. Must be Node v22+. Skip this = mystery failures.
2
Deploy to TEST
npm run deploy:test
Open test URL. Sign in as real user. Click the feature. Check console. Test mobile.
3
Get approval (yourself or teammate)
"I verified X, Y, Z on test. Approving for production." — then and only then.
4
Deploy to PRODUCTION
npm run build    npx wrangler pages deploy dist --project-name your-app
5
Verify on production
Same checks as test. Verify nothing else broke. Then it's done.
✕
NEVER: Deploy to prod without testing. Trust "it works". Bundle multiple changes.

Afternoon Session — 1:00 PM

Building Real Features — The 5 Rules

🔍

Research First, Always

Search project knowledge. Read CLAUDE.md. Find existing patterns. This rule would have saved 7 hours on a routing problem that was already documented.

1️⃣

One Thing At a Time

Never bundle multiple changes in one deploy. One feature, one brief, one deploy. Verify it works. Then move to the next.

✅

Test Then Prod — No Exceptions

"It worked in test" is the only green light for production. Never deploy directly to prod to save time.

👁

Verify In The Real Environment

Open the browser. Sign in. Click the thing. Read the console. Only what you verify with your own eyes counts as done.

🚫

No Fallbacks, No Placeholder Data

If data is missing, fail loudly. Never substitute "Your Business" or "Venture Clarity" for real data. Silent failures hide real bugs.

Afternoon Session — 1:45 PM

When Things Break — The Debugging Brief

STEP 1 — DO NOT FIX ANYTHING YET Read the exact error message. Paste it here: [error text] Read the file where the error occurs. Show me lines X–Y. STEP 2 — TRACE THE CODE PATH What function called what function caused this error? Show me the exact call chain. STEP 3 — GREP BEFORE FIXING Search the entire codebase for similar patterns. How many places have the same potential issue? STEP 4 — ONLY THEN FIX Fix the root cause, not the symptom. Show me the diff before applying it. Deploy to test and verify the error is gone.
💡

The most expensive debugging session (7+ hours) was caused by skipping Step 1. We guessed at solutions and eventually found a single routing config line — already documented in CLAUDE.md. Research before fix. Every time.

Appendix B

The 12 Biggest Mistakes

M01

Skipping research and guessing at solutions

Fix: Read CLAUDE.md first. No exceptions.

M02

Deploying to production without testing

Fix: Test → Verify → Approve → Prod. Always.

M03

Trusting AI's "it's done" without verifying

Fix: Open browser. Click the thing. Then it's done.

M04

Writing code instead of briefs

Fix: Outcomes, not implementations. 2 sentences.

M05

Exposing API tokens in chat

Fix: .env files only. Rotate immediately if exposed.

M06

Silent fallbacks masking real failures

Fix: No placeholder data. Fail loudly.

M07

Dynamic routes where static routes required

Fix: Know your framework's routing model cold.

M08

Bundling multiple changes in one deploy

Fix: One change, one deploy, one verification.

M09

Duplicate environment variables

Fix: Grep for dupes after every env change.

M10

Using bash instead of PowerShell on Windows

Fix: Document your OS in CLAUDE.md.

M11

Reporting preview URLs instead of canonical

Fix: Only two URLs matter: test and prod.

M12

Working past exhaustion

Fix: Fresh 2-hour session beats exhausted 5-hour session.
