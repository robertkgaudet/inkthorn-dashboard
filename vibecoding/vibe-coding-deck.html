Module M-02 — How the Web Works

How a Website Actually Works

Every time you visit a website, a small miracle happens in under a second. Here's exactly what's happening behind the scenes.

🧑‍💻
Step 1
You Type a URL
You type textos.ai — a human-readable address, like a street address for a computer.
→
📖
Step 2
DNS Lookup
DNS translates the name to an IP address (104.21.8.43). The internet's phone book — names to numbers.
→
🌐
Step 3
Request Travels
Your request hops across the internet in milliseconds, reaching the nearest Cloudflare server.
→
🖥️
Step 4
Server Responds
A web server — a computer that never turns off — sends back files: HTML, CSS, and JavaScript.
→
🎨
Step 5
Browser Renders
Your browser reads those files and paints what you see. Under 200ms, start to finish.
HTML — Structure

The skeleton of the page. Defines what's on it: headings, paragraphs, buttons, images. Like the walls and rooms of a building.

CSS — Style

The paint and furniture. Colors, fonts, spacing, layout, animations. CSS makes things look the way they look.

JavaScript — Behavior

The light switches and elevators. Handles clicks, loads data, updates content — makes the page interactive.

Module M-03 — The Full Stack

Every Layer of a Modern Web App

A real application has six distinct layers. Each does a specific job. When you understand them, you can direct AI to build any of them precisely.

🎨
UX / Interface Layer
What users see and click — HTML structure, CSS styling, JavaScript behavior. The only layer users experience directly.
Astro
🗺️
Routing Layer
Maps URLs to pages. When a user visits /dashboard, routing says "show them the dashboard." Without this, every URL shows a blank page.
Astro + _redirects
⚙️
Backend / API Layer
The brains — processes form submissions, calls AI, handles payments, enforces business rules. Invisible to users, essential to everything.
Cloudflare Workers
🔐
Authentication Layer
The bouncer — knows who you are and what you can see. Handles sign-up, login, logout, "forgot password," and keeps each user's data private.
Supabase Auth
🗄️
Database Layer
Permanent memory. Anything that must survive a page refresh — user data, posts, settings, transactions — lives here in structured tables.
Supabase / Postgres
🌍
Hosting / CDN Layer
Where your files live and how they reach users globally. CDN = Content Delivery Network. Copies of your app sit near every user on earth.
Cloudflare Pages

Module M-04 — Your Toolset

Supabase — Your Database & Login System

Think of Supabase as two things: a spreadsheet in the cloud that thousands of users can read and write simultaneously, plus a complete login system — managed through a visual dashboard.

🗄️ The Database

PostgreSQL — Industry Standard

The same database used by Instagram, Reddit, Shopify, and Airbnb. Battle-tested for 30+ years. Supabase wraps it with a visual table editor that looks like Google Sheets — see your data without writing any code.

🔐 Authentication

Complete Identity System, Built In

Sign up, sign in, sign out, "forgot password," magic links, and social login (Google, GitHub, Apple) — all ready to go. Without Supabase, building just the login takes weeks. With it, an afternoon.

🔒 Row Level Security

Each User Sees Only Their Data

A database rule that enforces privacy automatically. User A's data is invisible to User B — not because of code you wrote, but because the database itself enforces it. This is how multi-user apps work safely.

⚡ Auto-Generated API

Your Database Becomes an API Instantly

Create a table in Supabase and a REST API for that table is automatically generated. Your app can read, write, update, and delete data with no server code required on your end.

💡

The big picture: Supabase = "who is this user and what data belongs to them." Cloudflare = "how does the app reach the user." Together, they replace what used to require a full-time DevOps engineer.

Morning — M-05 · The Most Important Concept

The Two Roles — Architect vs. Implementer

Claude.ai — Your Role
The Architect
  • Write briefs specifying WHAT to build and WHY
  • Define constraints and patterns to follow
  • Reference existing working code as the pattern
  • Make all product decisions
  • Decide when something is good enough to ship
🚫 Never Write specific function bodies, line-by-line fixes, or paste code blocks in your briefs.
Claude Code — AI's Role
The Implementer
  • Figures out HOW to implement your brief
  • Writes and tests the actual code
  • Deploys to the test environment first
  • Reports what was done and what to verify
  • Never makes product decisions on its own
🚫 Never Implement anything without understanding the existing architecture first.

Morning — M-06

Writing Briefs That Actually Work

Good briefs produce good code. Vague briefs produce confused code. Rushed briefs produce wasted hours. This template eliminates all three.

WHAT: One sentence — the outcome, not the implementation "Add a live badge to the nav that pulses when a task is running" FILES: Exact file paths only — limits scope and prevents accidents src/pages/business/live.astro PATTERN: Reference existing working code in the project "Same animation as the pulsing dot in builder.astro around line 847" DO NOT: Explicit exclusions — prevents Claude Code touching the wrong files "Do not modify builder.astro, manage.astro, or any other page" VERIFY BY: How you will confirm it worked — in your own words "After test deploy, badge appears in nav and pulses when a task runs"
✅

2 sentences — not 30 lines of code. Describe outcomes, not implementations. That is the entire skill.

Module M-07 — First Deploy

Two Environments — Never Mix Them

Test Environment
your-app-test.pages.dev
  • All deployments go here first — no exceptions
  • No real charges, no real user data
  • Stripe in test mode (fake payment cards)
  • This is where you verify before going live
  • Only your test account can access it
Production Environment
app.yourdomain.com
  • Only after test environment is fully verified
  • Real users, real data, real transactions
  • Requires your explicit approval every time
  • Never deploy here without testing first
  • If it breaks here, real users are affected
🔥

A single typo in a config file took down production for 2 hours. The fix was one line. The lesson: test first, every single time.

Afternoon — A-03

The Deploy Ritual — Follow This Order Every Time

1
Pre-deploy authentication check
npx wrangler whoami    node --version
Confirms your API token is loaded. Node must be v22+. Skip this and you get mystery failures.
2
Deploy to the TEST environment
npm run deploy:test
Open the test URL. Sign in. Click the feature. Check the console. Test at mobile width (390px).
3
Give explicit approval to go to production
"I verified X, Y, and Z on test. Approving for production." — Write it down. Only then proceed.
4
Deploy to PRODUCTION
npm run build && npx wrangler pages deploy dist --project-name your-app
5
Verify on production — same checks as test
Confirm the feature works AND nothing else broke. Then it's done.
✕
Never: deploy to prod without testing · trust AI's "it works" · bundle multiple changes

Afternoon — A-01

Building Real Features — The 5 Non-Negotiable Rules

🔍

Research First, Always

Read CLAUDE.md and search project knowledge before writing any brief. This rule alone would have saved 7 hours on a routing problem that was already documented.

1️⃣

One Change at a Time

Never bundle multiple features in one deploy. One feature, one brief, one deploy. Verify it works. Then move to the next.

✅

Test Before Production — Always

"It worked in test" is the only acceptable green light for production. Never deploy directly to prod to save time.

👁

Verify With Your Own Eyes

Open the browser. Sign in. Click the thing. Read the console. Only what you see with your own eyes counts as done.

🚫

No Fallbacks, No Placeholder Data

If data is missing, fail loudly. Never substitute "Your Business" for a real business name. Silent failures hide real bugs and mislead real users.

Afternoon — A-02

When Things Break — The Debugging Brief Formula

STEP 1 — DO NOT FIX ANYTHING YET Read the exact error message. Paste it here in full: [error text] Read the file where the error occurs. Show me lines X through Y. STEP 2 — TRACE THE CODE PATH What function called what function that caused this error? Show me the exact call chain, step by step. STEP 3 — SEARCH BEFORE FIXING Search the entire codebase for the same pattern. How many places have the same potential problem? STEP 4 — ONLY THEN APPLY A FIX Fix the root cause — not the symptom. Show me the exact changes (diff) before applying anything. Deploy to test and verify the error is completely resolved.
📖

Real story: A 7-hour debugging session was caused by skipping Step 1. We guessed at solutions and deployed 15+ times. The answer was a single line in a config file — already documented in CLAUDE.md. Research before fix. Every time.
