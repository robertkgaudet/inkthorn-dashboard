Module 01

What Is Vibe Coding?

The complete shift in how software gets built — and why it changes everything for non-technical founders.

Vibe coding is using AI to build production software by describing what you want, not how to build it. Instead of writing code, you write instructions. Instead of being the engineer, you become the architect.

"I built TextOS in 5 days — sitting under an oak tree at lunch with my wife using only my laptop and Claude. No app. Just text. That's exactly what I'm selling."

Rob Gaudet, Founder of TextOS

The reason vibe coding works isn't magic. It works because AI is now skilled enough to write production code — and your job shifts from being the one who writes code to being the one who knows what to build and why. That's a fundamentally different skill set, and it's one you already have.

Why This Works Now (And Didn't 2 Years Ago)

2022

The Old Way

  • Hire a developer or learn to code yourself
  • Spend months on basic infrastructure
  • $50K–$200K before first user
  • Technical debt from day one
  • Ideas die in the gap between vision and execution
2026

The Vibe Coding Way

  • You describe the outcome. AI writes the code
  • Production infrastructure in hours
  • $0–$500 before first user
  • Architecture stays in your head + spec docs
  • Build speed limited by clarity, not code
⚠ Important Distinction

Vibe coding doesn't mean "ask AI to build everything and hope." It means you become the architect who directs the AI. Your judgment, taste, and product thinking is the most valuable part of the equation. AI handles the implementation. You handle the vision and the decisions.

The 3 Things That Make Vibe Coding Work

01

Architectural Thinking

You must understand WHAT you're building well enough to describe it. The more clearly you think, the better the output.

02

Research Before Code

Always understand the existing system before changing it. The biggest mistakes come from skipping research and guessing.

03

Verify Everything

AI claims it's done. Often it isn't. Always verify in the browser, logs, and database — not just by reading the AI's response.

Authentication Setup

Every tool needs credentials. This is the part most people skip and then wonder why nothing deploys. Set this up properly once and you'll never fight it again.

Cloudflare API Token — Setup Steps 
1. Go to: cloudflare.com → My Profile → API Tokens
2. Click "Create Token"
3. Use "Edit Cloudflare Workers" template
4. Set permissions: Workers Scripts (Edit), Pages (Edit), Account Settings (Read)
5. Copy the token immediately — you only see it once
6. Save to a password manager (Bitwarden, 1Password, etc.)
7. NEVER paste it in a chat, email, or document

// Store in your project's .env file like this:
CLOUDFLARE_API_TOKEN=cfut_xxxxxxxxxxxxxxxxxxxx

// Set it in PowerShell before deploying:
$env:CLOUDFLARE_API_TOKEN = (Get-Content ".env" | Where-Object { $_ -match "^CLOUDFLARE_API_TOKEN=" } | ForEach-Object { $_ -replace "^CLOUDFLARE_API_TOKEN=", "" })
🔴 Critical Security Rule

NEVER paste API tokens, secrets, or passwords into Claude.ai chat — even test-mode keys. Once in chat history, they can be accessed or exposed. Store credentials in a password manager and .env files only. If you accidentally expose a token, rotate it immediately in the provider's dashboard.

The Two Roles

This is the most important concept in the entire course. Get this right and everything else follows.

Your Role

The Architect

You describe outcomes, constraints, patterns to follow, and how to verify success. You make all product decisions. You define what done looks like.

  • Write briefs that specify WHAT to build and WHY
  • Define which files to touch and which to leave alone
  • Reference existing working code as the pattern
  • Decide when something is good enough to ship
  • Catch mistakes before they become expensive
Claude Code's Role

The Implementer

Claude Code figures out HOW to implement your brief. It writes the actual code, runs tests, deploys to the right environment, and reports results.

  • Figures out the implementation details
  • Writes and tests the actual code
  • Deploys to test environment first
  • Reports what was done and what to verify
  • NEVER makes product decisions autonomously
🔴 The Most Common Mistake

Writing code in your briefs. When you write specific function bodies, variable names, or copy-paste code blocks in your briefs, you become the implementer — and you're worse at implementation than Claude Code. Your value is in architectural direction, not code syntax. Write outcomes, not code.

✅ What a Good Brief Looks Like

"Add a bulletin feed that reads from task_runs, deduplicates by task slug, shows 8 most recent events as human-readable sentences, newest first. Follow the same pattern as the existing renderTasks() function in builder.astro." — That's 2 sentences. Not 30 lines of JavaScript.

Writing Briefs That Work

A brief is your instruction to Claude Code. Good briefs produce good code. Vague briefs produce confused code. Rushed briefs produce wasted hours.

The Anatomy of a Good Brief

Brief Template 
WHAT: [One sentence describing the outcome, not the implementation]
  Example: "Add a live 'LIVE' badge to the nav that pulses green 
  when any task is running for this business"

FILES TO TOUCH: [Exact file paths — limits scope]
  src/pages/business/live.astro — add the badge HTML and CSS

PATTERN TO FOLLOW: [Reference existing working code]
  Same animation pattern as the existing pulsing dot in 
  builder.astro around line 847

DO NOT TOUCH: [Explicit exclusions prevent accidents]
  Do not modify builder.astro, manage.astro, or any other page

VERIFY BY: [How to confirm it worked]
  After deploying to test, confirm the badge appears in the nav 
  and pulses amber when tasks are running in the console

Building Real Features

The deploy cycle is the most important rhythm in vibe coding. Brief → Deploy Test → Verify → Approve → Deploy Prod.

1️⃣

Research First, Always

Before writing ANY brief, search project knowledge and read CLAUDE.md. Does a component already exist for this? Is there an established pattern? This single rule would have saved 7 hours on a routing problem that was already documented.

2️⃣

One Thing At a Time

Never bundle multiple changes in one deploy. If something breaks, you'll have no idea which change caused it. One feature, one brief, one deploy. Verify it works. Then move to the next.

3️⃣

Test Then Prod — No Exceptions

Deploy to test first. Verify with your own eyes in the browser. Only then approve production. "It worked in test" is the only green light for production. Never deploy directly to prod to save time.

4️⃣

Verify in the Real Environment

Claude Code saying "it works" means nothing. Open the browser. Sign in as a real user. Click the thing. Test at mobile width. Read the console. Check the database. Only what you verify with your own eyes counts as done.

5️⃣

No Fallbacks, No Placeholder Data

If your code is supposed to show a user's business name but the name isn't loaded yet, show a loading state — not "Your Business." Fake data in production trains users to ignore real data and hides real bugs. If it's missing, show it's missing.

When Things Break

Things will break. Every single time. The skill isn't avoiding breaks — it's diagnosing and fixing them faster than anyone else.

The Debugging Brief Formula

Debugging Brief — Always Start Here 
STEP 1: DO NOT FIX ANYTHING YET
Read the exact error message. Paste it here: [error text]
Read the file where the error occurs. Show me lines [X-Y].

STEP 2: TRACE THE CODE PATH
What function called what function that caused this error?
Show me the exact call chain.

STEP 3: GREP BEFORE FIXING
Search the entire codebase for similar patterns.
How many places have the same potential issue?

STEP 4: ONLY THEN FIX
Fix the root cause, not the symptom.
Show me the diff before applying it.
Deploy to test and verify the error is gone.
💡 Lesson from the Real Build

The most expensive debugging session (7+ hours) was caused by skipping Step 1. We guessed at solutions, deployed 15+ times, and eventually found that a single line in a routing config file — already documented in CLAUDE.md — was the entire problem. Research before fix. Every time.

Shipping to Production

The deploy ritual that keeps production healthy. Do this in order every single time.

The Deploy Ritual — Non-Negotiable Order 
STEP 1: Pre-deploy check
npx wrangler whoami   ← confirms auth token is loaded
node --version        ← must be v22+

STEP 2: Deploy to TEST
npm run deploy:test
→ Verify at: https://your-app-test.pages.dev
→ Sign in as real user
→ Test the feature you changed
→ Check browser console for errors
→ Test at mobile width (390px)

STEP 3: Get approval (yourself or a teammate)
→ "I verified X, Y, Z on test. Approving for production."

STEP 4: Deploy to PRODUCTION
npm run build
npx wrangler pages deploy dist --project-name your-app

STEP 5: Verify on production
→ Open https://app.yourdomain.com
→ Verify the same things you verified on test
→ Check that nothing else broke

NEVER: Deploy directly to production without testing
NEVER: Trust Claude Code's "it works" claim without verifying yourself
NEVER: Deploy multiple changes in one batch
Appendix A

15 Lessons From The Real Build

Every lesson below was learned the hard way during 3 weeks building TextOS. You don't have to repeat them.

⚡

Lesson 1: Research is the job

Before touching any code, search for existing patterns. The answer is almost always already in the codebase. A 7-hour debugging session was solved by reading a config file that had the answer documented. Research first. Always.

🏗

Lesson 2: Architecture lives in CLAUDE.md

Every architectural decision you make needs to live in CLAUDE.md. The routing pattern, the deploy pattern, the environment variables, the roles. If it's not in CLAUDE.md, a new session won't know it and will repeat your mistakes.

✅

Lesson 3: AI claims "done" before it's done

Claude Code will tell you something is deployed and working. Often it isn't. The deploy failed silently, or it deployed to a hash preview URL instead of the real URL, or the feature has a subtle bug. Verify everything yourself.

🔒

Lesson 4: Tokens get exposed in the worst moments

API tokens were accidentally displayed in plain text in chat twice during the build. Both required immediate rotation. Never ask AI to show you a token, never paste a token in chat, never echo a token value. Read tokens from files silently.

🧠

Lesson 5: Silent failures are worse than crashes

Code that silently falls back to placeholder data caused hours of confusion. A business was getting named "Venture Clarity" instead of its real name — and no error was thrown. Make your code fail loudly. A crash you can find. A silent fallback you can't.

🎨

Lesson 6: Design with the user in mind from day one

We spent a full session building a design vision before touching any code for the Business Manager page. That vision conversation took 2 hours but saved 20 hours of code-first iterations. Get the design right on paper first.

📱

Lesson 7: Mobile first, always

Every page must be tested at 390px (iPhone 14 Pro) before it ships. Twice. Once when you build it, once after every deploy. Mobile is how most users will first see your product. If it doesn't work on mobile, it doesn't work.

🔄

Lesson 8: Session context resets — plan for it

Every new Claude.ai session starts fresh. Your CLAUDE.md in project knowledge is what carries context forward. The morning ritual of syncing project knowledge and asking for overnight ideas is what keeps the session productive from minute one.

⛔

Lesson 9: Never make the founder the bottleneck

If every deploy requires you to manually paste a command, you've become the bottleneck. Invest upfront in making deployments autonomous. If Claude Code can't deploy on its own, that's a tooling problem to fix, not a workflow to accept.

📝

Lesson 10: Data-driven beats hardcoded

Every hardcoded list in your code is future technical debt. When tasks were hardcoded in an array instead of the database, adding one task required a code deploy. Make things data-driven from day one — your future self will thank you.

🚀

Lesson 11: Speed comes from clarity, not typing

The fastest sessions were the ones where the brief was written carefully before any code was touched. The slowest sessions were where we jumped straight to coding without a clear outcome in mind. Clarity is the multiplier.

🔍

Lesson 12: Wrangler auth fails silently

The deployment tool (Wrangler) can silently fail to authenticate. Every deploy brief must start with npx wrangler whoami to confirm auth. If you skip this, you'll think something deployed when it didn't.

💰

Lesson 13: Model choice is a business decision

Claude Sonnet for quality, Claude Haiku for speed and cost. Using Sonnet for everything burns budget fast. Using Haiku for everything produces mediocre output. Route tasks by value: high-value user-facing content gets Sonnet, routine tasks get Haiku.

⚠

Lesson 14: Duplicate env vars will wreck you

A duplicate variable in .env.production (one correct, one a leftover placeholder) took down production. The second entry silently overrides the first. Always check your env files for duplicates after adding anything new.

🎯

Lesson 15: 30 years of architecture compounds at AI speed

The reason TextOS was built in 5 days isn't just AI — it's 30 years of knowing which patterns are right, which shortcuts create debt, and which decisions matter. Vibe coding amplifies your expertise. The clearer your thinking, the better the output.

Appendix B

The 12 Biggest Mistakes

Every mistake on this list happened during the real TextOS build. Some cost hours. Some nearly cost the launch.

🚨

Mistake 1: Skipping research and guessing at solutions

The routing architecture for the entire frontend was documented in a config file and a CLAUDE.md entry. Ignoring those docs and guessing led to 7+ hours of failed deployment attempts.

Fix: Read CLAUDE.md and search project knowledge BEFORE writing any brief. No exceptions.
🚨

Mistake 2: Deploying to production without testing

Skipping the test environment to save time caused a production outage. The fix was trivial — but real users couldn't sign in while it was broken.

Fix: Test → Verify → Approve → Prod. This order is non-negotiable. Speed isn't worth a production outage.
🚨

Mistake 3: Trusting AI's "it's done" without verifying

Multiple times, Claude Code declared a feature complete when the deploy had failed, the feature had a bug, or the wrong URL was being tested.

Fix: Open the browser. Sign in. Click the thing. Read the console. What you see with your own eyes is done. Nothing else is.
🚨

Mistake 4: Writing code instead of briefs

When the architect starts writing specific code in briefs, they create implementation bugs that Claude Code faithfully copies. A 30-line JavaScript function in a brief had a scope bug that took hours to diagnose.

Fix: Describe outcomes, not implementations. 2 sentences, not 30 lines of code.
🚨

Mistake 5: Exposing API tokens in chat

Two separate times, Cloudflare API tokens were accidentally displayed in plain text in chat and required immediate rotation. One was caused by asking Claude Code to show the token value to confirm it was loaded.

Fix: Tokens live in .env files. Read them silently. Show first 4 chars max to confirm. Rotate immediately if exposed.
🚨

Mistake 6: Silent fallbacks masking real failures

A missing input field caused the AI to receive null data and respond with generic business names ("Venture Clarity") instead of the user's real business name. No error was thrown, so the bug went undetected.

Fix: If data is missing, throw an error and halt. Never substitute placeholder data for real data silently.
🚨

Mistake 7: Using dynamic routes where static routes are required

The framework (Astro on Cloudflare Pages) has specific routing rules that weren't followed, causing pages to redirect to the homepage instead of loading. The rule was in CLAUDE.md the entire time.

Fix: Understand your framework's routing model before creating any new page. Document it in CLAUDE.md so you never forget.
🚨

Mistake 8: Bundling multiple changes in one deploy

Deploying 5 features at once made it impossible to tell which one broke production when something went wrong. The rollback and re-deploy cycle cost more time than deploying them separately would have.

Fix: One change, one deploy, one verification. Then move to the next. The overhead is worth it every time.
🚨

Mistake 9: Duplicate environment variables

A leftover placeholder URL in .env.production silently overrode the correct URL. Production pointed at a non-existent server and all API calls failed.

Fix: After every env file change, grep for duplicates. Two entries with the same key = guaranteed to be wrong.
🚨

Mistake 10: Using bash instead of PowerShell on Windows

Claude Code defaulted to bash commands which don't work on Windows, causing deploy commands to fail with cryptic path errors ("C:codetextos-web: No such file or directory").

Fix: Document your OS in CLAUDE.md. Windows uses PowerShell with backslash paths. Never assume bash.
🚨

Mistake 11: Reporting preview URLs instead of canonical URLs

Every Cloudflare Pages deploy creates a hash preview URL (abc12345.pages.dev) AND updates the canonical URL (yourapp.pages.dev). Claude Code kept reporting the hash URL, leading to confusion about whether the canonical URL was actually updated.

Fix: Only two URLs matter: test canonical and production. Hash preview URLs are irrelevant — never report them.
🚨

Mistake 12: Working past exhaustion

Sessions that ran past midnight produced the most bugs and the most wasted time. A fresh 2-hour morning session consistently accomplished more than an exhausted 5-hour night session.

Fix: When you're exhausted, stop. Commit your work, write a handoff note, sleep. Fresh eyes fix in minutes what tired eyes miss for hours.
