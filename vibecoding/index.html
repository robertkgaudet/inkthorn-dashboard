CLAUDE vs CLAUDE CODE — Role Definitions

The most important rule in AI-assisted development: keep the architect and the implementer separate. Claude thinks and plans. Claude Code builds and ships.

CLAUDE
claude.ai — The Architect
  • Write briefs that specify WHAT to build and WHY
  • Define constraints and patterns to follow
  • Reference existing working code as the pattern
  • Define how to verify success
🚫 Never Write specific function bodies, line-by-line fixes, or copy-paste code blocks
CLAUDE CODE
claude code CLI — The Implementer
  • Figure out HOW to implement the brief
  • Write the actual code
  • Run, test, and verify the implementation
  • Deploy and report results
🚫 Never Implement without understanding the existing architecture first

Protocol — Run Once Per 24 Hours

Daily Architecture Review

At the start of each day or new session, Claude Code must orient itself before touching a single file.

1
Read CLAUDE.md in both repos
textos-web and textos-agent — both, every time
2
Review key codebase areas for changes
Components, scripts, layouts, API patterns, Supabase queries, auth patterns, nav, _redirects, env vars
3
Ask the five questions before writing any code
Does a component exist? A function? What's the routing pattern? Are there shared styles? Is there an existing data access pattern?
4
When uncertain — READ the working implementation first
Never assume. Always verify.

Non-Negotiable Protocol

Research Before Code

1
Search for existing patterns
Does a similar feature already exist? Read it fully before writing anything new.
2
Understand architecture constraints
Read public/_redirects before creating new pages. Read wrangler.toml before touching worker config.
3
Identify reusable elements
Components, functions, styles, API calls, auth patterns. If it exists, use it. Never reinvent.
4
Only THEN write code
Implementing without researching = wasted time. Rob's time is the most valuable resource.

Architecture Rules — Memorize These

Key Architecture Rules

Routing — Non-Negotiable

All /business/{slug}/* pages use static prerendered pages + _redirects rewrites + slug from localStorage
NEVER use dynamic [slug] routes under /business/
NEVER use prerender = false under /business/
NEVER use _routes.json includes for /business/* patterns

Script Pattern for Standalone Pages

Supabase CDN via is:inline script in <head>
Env vars via define:vars block → expose as window globals
Main script as plain <script> (not type="module")
No ES module imports inside define:vars blocks

Deploy Pattern

Always test first: npm run deploy:test
Verify at https://textos-web-test.pages.dev before touching prod
Never deploy to prod without Rob's explicit approval
